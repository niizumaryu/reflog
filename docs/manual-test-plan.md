# 手動テスト計画(開発経験の少ない人向け)

このドキュメントは、開発経験がなくてもREFLOGの主要機能を手動で確認できるようにするための手順書です。各項目について、上から順に実施し、結果を「合格」「不合格」欄に記録してください。不合格の場合は「失敗時に撮影すべき画面」に記載した画面のスクリーンショットを残してください。

準備:
- 開発環境の場合は `npm run dev` を実行し、`http://localhost:3000` を開く
- 本番環境の場合は実際の公開URLを開く
- テスト専用のメールアドレス(自分がすぐ確認できるもの)を1つ用意する

---

## A. 認証・プロフィール

### A-1. 新規登録

- 開くURL: `/login`
- 操作: 「アカウントをお持ちでない方は新規登録」を押し、テスト用メールアドレスとパスワード(6文字以上)を入力して「新規登録」を押す
- 期待結果: 「確認メールを送信しました」というメッセージが表示される。届いたメール内のリンクを開くとログイン状態になり `/` へ遷移する
- 失敗時に撮影すべき画面: エラーメッセージが出た画面、メールが届かない場合はメールの受信箱全体
- DB/Storage確認: 不要
- 合格 / 不合格: ______

### A-2. ログイン・ログアウト

- 開くURL: `/login`
- 操作: 登録済みのメールアドレス・パスワードでログイン→ログイン後、設定画面から「ログアウト」を押す
- 期待結果: ログイン後はトップ画面(`/`)が表示される。ログアウト後は `/login` に戻り、再度保護ページ(例: `/dashboard`)を直接開こうとすると `/login` にリダイレクトされる
- 失敗時に撮影すべき画面: ログイン後の画面、ログアウト後にリダイレクトされない場合はそのURLバー
- 合格 / 不合格: ______

### A-3. 未認証時のリダイレクト

- 開くURL: ログアウト状態で `/dashboard` を直接開く
- 期待結果: 自動的に `/login?next=%2Fdashboard` にリダイレクトされる
- 失敗時に撮影すべき画面: リダイレクトされなかった画面
- 合格 / 不合格: ______

### A-4. プロフィール未設定時の誘導

- 操作: 新規登録直後(ユーザー名未設定)の状態で `/dashboard` など任意のページを開く
- 期待結果: 自動的に `/settings/profile?onboarding=1` に遷移する
- 合格 / 不合格: ______

### A-5. パスワード再設定

- 開くURL: `/reset-password`
- 操作: 登録済みメールアドレスを入力して送信
- 期待結果: 届いたメール内のリンクから `/update-password` に遷移し、新しいパスワードを設定できる
- 失敗時に撮影すべき画面: エラー画面、メール未着の場合は受信箱
- 合格 / 不合格: ______

## B. スケジュール

### B-1. 新規作成〜一覧表示

- 開くURL: `/schedule/new`
- 操作: 大会名・日付を入力して保存
- 期待結果: 「予定を保存しました！」と表示され `/schedule` に戻ると、一覧に新しい予定が表示される
- DB確認: 必要であれば、Supabaseダッシュボードの Table Editor で `schedules` テーブルに新しい行があるか確認
- 合格 / 不合格: ______

### B-2. 編集・削除

- 開くURL: `/schedule/[id]/edit`(一覧から対象の予定を開く)
- 操作: 内容を変更して更新→続けて削除ボタンを押し、確認ダイアログで「OK」
- 期待結果: 更新内容が反映される。削除後は一覧から消える
- 合格 / 不合格: ______

### B-3. 長文入力

- 操作: メモ欄に2000文字を超える長文を貼り付けようとする
- 期待結果: 2000文字を超えて入力できない(`maxLength` により自動的に制限される)
- 合格 / 不合格: ______

### B-4. 他人の予定へのアクセス

- 操作: 自分の予定のURL(例: `/schedule/1234-...`)のIDを、別のテストアカウントでログインした状態で直接開く
- 期待結果: 「予定が見つかりませんでした」と表示される(他人のデータは見えない)
- DB確認: 不要(RLSにより自動的にブロックされる)
- 合格 / 不合格: ______

## C. 通知(Web Push — メールではありません)

### C-1. 通知の許可・購読

- 開くURL: `/settings`
- 操作: 「プッシュ通知」のトグルを押す→ブラウザの許可ダイアログで「許可」を選ぶ
- 期待結果: 「🟢 通知設定済み」の表示になる
- 合格 / 不合格: ______

### C-2. テスト通知

- 操作: C-1の後に表示される「テスト通知を送信」ボタンを押す
- 期待結果: 数秒以内にブラウザ通知が届く。「テスト通知を送信しました」と画面にも表示される
- 失敗時に撮影すべき画面: エラーメッセージ、ブラウザの通知許可設定画面
- 合格 / 不合格: ______

### C-3. 通知の解除

- 操作: トグルをもう一度押してOFFにする
- 期待結果: 「🔴 通知未設定」に戻り、以後テスト通知ボタンは表示されない
- DB確認: 可能であれば `push_subscriptions` テーブルから該当行が消えていることを確認
- 合格 / 不合格: ______

### C-4. 通知種別の設定

- 開くURL: `/settings/notifications`
- 操作: 各通知(前日リマインド・当日リマインド・未記録リマインド・月末振り返り・AIアドバイス)のトグルを個別にON/OFF
- 期待結果: それぞれ独立して切り替えられ、画面を再読み込みしても設定が保持される
- 合格 / 不合格: ______

## D. 動画分析(デモ解析であることの確認)

### D-1. デモ表示の確認

- 開くURL: `/video-analysis/new`
- 期待結果: 画面上部に「⚠️ デモ解析パイプライン」というオレンジ色の注意書きが常に表示されている
- 合格 / 不合格: ______

### D-2. アップロード〜解析

- 操作: MP4形式の短い動画(数秒〜数十秒)を選択してアップロード
- 期待結果: 「計測結果(実測値)」に動画の長さ・解像度が正しく表示される。「アップロードして解析を開始」を押すと、アップロード中は進捗パーセント表示とプログレスバーが表示される。しばらくすると処理中画面→解析結果画面に遷移する。結果画面にも「デモ」の注意書きがある
- 失敗時に撮影すべき画面: エラーメッセージ画面
- 合格 / 不合格: ______

### D-2.1. アップロードのキャンセル(2巡目監査で追加)

- 操作: 大きめの動画(数十MB以上、Wi-Fi環境推奨)を選択してアップロードを開始し、進捗表示が0〜100%の間にあるうちに「キャンセル」ボタンを押す
- 期待結果: アップロードが中断され、「アップロードをキャンセルしました。」と表示される。選択画面に戻り、同じ動画で再度アップロードをやり直せる
- Storage確認(可能であれば): Supabaseダッシュボード → Storage → `match-videos` バケットに、キャンセルしたはずのファイルが残っていないことを確認
- 合格 / 不合格: ______

### D-3. 不正な形式・サイズ超過

- 操作: 画像ファイル(.jpg等)を選択してみる。可能であれば300MBを超える動画も試す
- 期待結果: 「対応していない動画形式です」等のエラーが表示され、アップロードが実行されない
- 合格 / 不合格: ______

### D-4. 削除

- 開くURL: 解析結果一覧の詳細ページ
- 操作: 削除ボタンを押す
- 期待結果: 削除に成功し一覧から消える
- Storage確認: Supabaseダッシュボード → Storage → `match-videos` バケットで、該当ファイルが消えていることを確認
- 合格 / 不合格: ______

## E. アカウント削除(最重要)

**このテストは必ずテスト専用アカウントで実施してください。本番の実データが入ったアカウントでは絶対に実行しないでください。**

### E-1. 削除前の準備

- 操作: テストアカウントで、試合記録を1件、予定を1件、動画分析を1件(可能であれば)作成しておく
- 合格 / 不合格: ______(準備が整ったか)

### E-2. 削除の実行

- 開くURL: `/settings`
- 操作: 一番下の「アカウントを削除する」を押す→確認ダイアログで内容を確認し「削除する」を押す
- 期待結果: 「削除中...」の表示後、自動的に `/login` に遷移する(ログアウトされている)
- 失敗時に撮影すべき画面: エラーメッセージ全文
- 合格 / 不合格: ______

### E-3. データが本当に消えたかの確認(Supabaseダッシュボードで確認)

- 操作: Supabaseダッシュボード → Table Editor → `profiles` / `matches` / `schedules` / `video_analyses` の各テーブルで、削除したユーザーIDの行がないことを確認
- 操作: Supabaseダッシュボード → Storage → `match-videos` バケットと `profile-icons` バケットの両方で、削除したユーザーIDのフォルダが**存在しない**ことを確認(これが今回の監査で修正した最重要ポイントです)
- 操作: Supabaseダッシュボード → Authentication → Users で、削除したユーザーが一覧にいないことを確認
- 期待結果: すべて「存在しない」ことを確認できる
- 失敗時に撮影すべき画面: 削除したはずのフォルダ・行が残っている画面
- 合格 / 不合格: ______

## テスト結果の記録方法

各項目の「合格 / 不合格」欄に日付と結果を記入し、不合格だった項目は指定のスクリーンショットを添えて開発担当者へ共有してください。全項目合格になるまで、本番公開・ストア申請は見送ることを推奨します。
