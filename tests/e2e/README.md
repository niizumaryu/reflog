# E2E テスト (Playwright)

## 実行方法

```
npm run test:e2e        # ヘッドレス実行
npm run test:e2e:ui     # Playwright UIモード(デバッグ用)
```

既存の `next dev` プロセスがあればそれを再利用し、なければ `playwright.config.ts`
が自動起動します(Next.js 16はプロジェクトごとに1つの `next dev` しか許可しないため)。

## 2種類のテスト

### `tests/e2e/public/**`(常に実行される)

認証不要で検証できる範囲: 保護ルートへの未認証アクセスがログインへリダイレクトされる
こと、ログイン画面、404、利用規約・プライバシーポリシー、レスポンシブレイアウト(横スク
ロールなし・タップ領域44px以上)。追加の設定なしに常に4ビューポート(360×800 / 390×844 /
768×1024 / 1440×900)で実行されます。

### `tests/e2e/authed/**`(テスト用Supabaseプロジェクトが必要)

ホーム画面・クイック記録・スケジュールCRUD(ConfirmDialogでの削除を含む)・通知設定・
動画分析のデモ免責表示とサイズ超過エラーなど、ログインが必要な導線を検証します。

Supabaseの認証はサーバー側ミドルウェア(`src/proxy.ts`)が `supabase.auth.getUser()` で
実際にトークンを検証する設計のため、ブラウザ側のネットワークモック(Playwrightの
`page.route`)では再現できません。そのため、**本番コードに一切テスト用バイパスを追加せず**、
実際のUIからログインする方式(Playwrightの `storageState` パターン)を採用しています。

有効化する手順:

1. 本番とは別の、**テスト専用のSupabaseプロジェクト**を用意する(無料プランで可)。
2. そのプロジェクトに、使い捨てのテストアカウント(メール・パスワード)を1つ作成する。
3. `.env.local` の `NEXT_PUBLIC_SUPABASE_URL` / `NEXT_PUBLIC_SUPABASE_ANON_KEY` を
   そのテスト専用プロジェクトの値に切り替える(**本番プロジェクトを指したまま実行しない
   こと** — このテストは実際に試合記録・予定の作成/削除を行います)。
4. `.env.local` に以下を追加する。

   ```
   E2E_TEST_USER_EMAIL=e2e-test-account@example.com
   E2E_TEST_USER_PASSWORD=...
   ```

5. `npm run test:e2e` を実行すると、`E2E_TEST_USER_EMAIL` / `E2E_TEST_USER_PASSWORD`
   が両方設定されている場合のみ `auth-setup` プロジェクトと `authed-*` プロジェクトが
   登録され、実行されます。未設定の場合は登録自体されず(スキップ扱いにもならず)、
   `public-*` のみが実行されます — CIやこのリポジトリの標準環境で秘密情報なしに
   安全に実行できます。

## 既知の制約

- `authed/**` は本セッションでは実行確認できていません(テスト専用Supabaseプロジェクトが
  用意されていないため)。ロジックは `public/**` と同じ形式で作成済みですが、実際の
  Supabaseレスポンス形状に対する動作確認は未実施です。有効化後、最初の1回はUIモードで
  実行して確認することを推奨します。
- スクリーンショット回帰テストは今回追加していません(ベースライン画像の初回確定作業が
  必要で、不安定になりやすいため)。`screenshot: "only-on-failure"` により失敗時の
  スクリーンショットは自動保存されます。
